# AGENTS.md

本文件是 Agent 进入 SingBridge 仓库后的第一入口，用于约束 AI 在本项目中的默认开发行为。更具体的用户指令优先；如果规则冲突，先停止并向用户说明，不要自行选择其一继续实现。

## 必读顺序

进入仓库后，先读取：

1. `AGENTS.md`：本文件，AI 协作强约束。
2. `foreign-song-singalong-prd.md`：产品目标、范围边界和验收标准。
3. `foreign-song-singalong-technical-design.md`：架构、数据流、技术栈和关键技术决策。
4. `foreign-song-singalong-task-breakdown.md`：里程碑、任务拆解和验证命令。
5. `japanese-reading-technical-spike.md`：日语 tokenizer spike 结论；涉及日语读音适配时必须读取。
6. `apps/web/Claude_DESIGN.md` 和 `apps/web/Spotify_DESIGN.md`：涉及 WebUI 时必须读取；Claude 为主风格，Spotify 只用于状态、高亮和异常。

当前仓库没有根目录 `README.md`，也没有 `docs/` 目录；不要套用通用模板里的默认路径。不要跳过项目文档直接编码。

## 单一事实来源

- AI 行为约束、授权边界和错误恢复流程以 `AGENTS.md` 为准。
- 产品定位和验收边界以 `foreign-song-singalong-prd.md` 为准。
- 技术架构、数据模型和依赖边界以 `foreign-song-singalong-technical-design.md` 为准。
- 任务拆解、里程碑和验证口径以 `foreign-song-singalong-task-breakdown.md` 为准。
- 日语 tokenizer 是否替换，以 `japanese-reading-technical-spike.md` 和当前 fixture 证据为准。
- WebUI 视觉方向以 `apps/web/Claude_DESIGN.md` 为主；`apps/web/Spotify_DESIGN.md` 只用于当前状态、异常和校正高亮。

如果这些文件互相冲突，先报告冲突，不要自行选择其一继续实现。

## SingBridge 项目边界

- MVP 只做日语。
- 用户提供歌词；不做歌词搜索、公开歌词库或云端同步。
- CLI first；WebUI 只消费 CLI 生成的 annotation JSON 和 correction draft JSON。
- WebUI 不直接接收原始歌词，不在浏览器跑 tokenizer。
- 不做音频播放、录音、评分、账号、社区、移动 App 或复杂编辑器。
- `kuromoji` 暂时保留，并隔离在 `packages/core/src/japanese/readingAdapter.ts` 后面。
- 不从 romaji 自动反推 kana；`suggestedKana` 在 correction draft 中保持 `null`，除非后续有明确人工校验流程。
- 不覆盖已有 manual override，除非任务和设计明确要求。
- 真实歌曲完整歌词只放本地忽略目录，例如 `.local-song-validation/`；仓库样例只保留短片段或合成文本。

## 开发前原则

在实现前必须明确：

- 本次目标是什么。
- 哪些行为被视为完成。
- 哪些行为明确不做。
- 输入、输出、边界和失败情况是什么。
- 如何验证，验证命令是什么，预期结果是什么。

如果目标存在多种解释且实现差异较大，必须先询问用户。询问时给出可选方案、差异和代价，不要只问“你想怎么做”。

## 默认开发顺序

非简单任务默认按以下顺序推进：

```text
需求拆解 -> 流程图 -> 技术架构 -> 技术栈选择 -> 数据结构 -> CLI -> 自动化测试 -> 小样本验证 -> WebUI
```

规则：

- 需求不清楚时，不编码。
- 先 CLI 后 WebUI。
- 先本地最小闭环，后外部服务集成。
- 先小样本验证，后扩大数据范围。
- 每个阶段都要有可复制的验证命令和预期结果。

## 需求拆解要求

每个功能点至少写清：

- 功能目标。
- 输入。
- 输出。
- 业务边界。
- 异常情况。
- 可测试条件。
- 自动化测试方案；无法自动化时，写人工验证方案。

不接受不可验证的任务描述，例如“优化体验”“补齐能力”“增强稳定性”，除非已经拆成具体行为和验收标准。

## 架构与选型要求

正式实现前，技术架构至少说明：

- 运行形态：CLI、本地脚本、Web 应用、后台服务、桌面应用、定时任务等。
- 模块边界：输入层、处理层、存储层、输出层、外部集成层。
- 数据流：数据从哪里来，经过哪些处理，最终写到哪里或展示给谁。
- 关键依赖：数据库、文件、第三方 API、模型服务、浏览器、队列、缓存等。
- 部署方式：本地运行、单机部署、云端部署或混合部署。
- 失败处理：外部依赖不可用、数据缺失、权限不足、网络失败时如何降级。

如果技术栈未确定，必须给出推荐方案、备选方案、明确不选方案、各方案优缺点和当前阶段选择理由。

选型偏好：

- 优先低成本、低运维、可本地验证。
- 优先使用当前项目已有技术和风格。
- 优先用清晰边界、简单数据结构、可读配置和测试保留扩展性。
- 不为假想规模提前引入插件系统、多租户、多数据库后端、复杂权限或分布式架构。

## 数据与存储原则

- 能不持久化就不持久化。
- 能用文件就先用 JSON、YAML、Markdown、CSV。
- 需要本地查询或分析时，优先 SQLite 或 DuckDB。
- 需要多人协作、服务化或强一致性时，再考虑 PostgreSQL、MySQL 或云数据库。
- 云服务和远程数据库不应成为早期核心流程的唯一依赖，除非需求明确要求。
- 数据模型只服务当前确认的功能，不为未来假想场景提前复杂化。

## CLI 优先原则

CLI 应具备：

- 清晰命令名。
- `--help` 可读。
- 参数含义明确。
- 输出可复制、可测试。
- 成功和失败行为明确。
- 失败时返回非 0 退出码。
- 错误提示包含原因和下一步建议。

WebUI 只能在以下条件满足后推进：

- 核心 CLI 流程稳定。
- 数据结构稳定。
- 用户操作路径已验证。
- 错误和边界情况已清楚。
- 已有足够样例数据支撑界面设计。

## 任务分级

### 只读核查

允许：

- 读取文件、搜索文本、查看目录。
- 查看 `git status`、`git diff`、`git log`、`git show`。
- 读取日志、配置键名、CSV、JSON、Markdown、代码片段。
- 执行数据库 `SELECT`。
- 调用不会改变状态的查询接口。

不得：

- 写文件。
- 写数据库。
- 执行迁移。
- 安装、升级或移除依赖。
- 启动、停止或重启非本地开发服务。
- 提交、拉取、推送或合并 Git。

### 小改动

允许：

- 修改当前任务直接相关的代码、测试或文档。
- 运行本地匹配范围的测试、lint、type check。
- 启动本地开发服务用于自测，仅限本地端口，不绑定 `0.0.0.0`，不暴露公网。

不得：

- 写真实业务数据。
- 执行生产或测试环境迁移。
- 调用真实业务写接口。
- 引入未授权依赖。
- 扩大产品范围。

### 真实联调

以下行为属于真实联调：

- 调用真实接口。
- 写数据库。
- 执行 worker 或定时任务。
- 浏览器自动化端到端验证。
- 跨系统验证。
- 服务状态检查。
- 任何可能改变外部系统状态的操作。

执行前必须说明环境、对象、影响范围、验证证据和回退或停止方案。

## 授权边界

必须得到明确授权才能执行：

- `git add`、`git commit`、`git push`、`git pull`、`git rebase`、`git merge`。
- 删除文件或目录。
- 写数据库、执行迁移、数据修复、软删除、回滚。
- 安装、升级、移除依赖。
- 修改环境变量或运行配置。
- 调用真实业务写接口。
- 启动、停止、重启非本地开发服务。
- 发布到测试、预发或生产环境。
- 任何会改变线上状态的操作。

默认允许：

- 读取文件和搜索文本。
- 查看目录、状态和 diff。
- 运行项目自带的只读检查。
- 运行本地测试、lint、type check。
- 启动本地开发服务自测。
- 做方案比较、风险分析和文档整理建议。

## 修改纪律

- 修改前说明要改哪些文件和目标。
- 每一行改动都应能追溯到当前需求。
- 只改当前任务直接相关文件。
- 不做无关重构、无关格式化、无关命名调整。
- 不扩大接口能力。
- 不引入未请求的配置化、插件化、多后端适配。
- 匹配项目现有风格，即使有更偏好的写法。
- 如果发现无关死代码，只说明，不删除。
- 自己引入的无用代码、无用导入和临时兼容逻辑必须清理。

## 简单优先

- 用最少代码解决已确认问题。
- 不为单次使用创建抽象。
- 不为不可能发生的场景写复杂错误处理。
- 不添加用户没有要求的功能。
- 如果实现明显可以从 200 行变成 50 行，应优先简化。

## 不确定性处理

遇到疑问时：

1. 先查项目文档、现有代码、测试和数据样例。
2. 如果能在项目内闭环，说明判断依据。
3. 如果不能闭环，立即向用户确认。

不得静默猜测：

- 产品方向。
- 数据含义。
- 工作流假设。
- 用户意图。
- 高风险技术选型。
- 删除、覆盖、迁移或写入范围。

## 测试与验收

新增功能必须有测试或明确验证命令。

优先级：

1. 自动化测试。
2. 集成测试或 CLI 行为验证。
3. 小样本端到端验证。
4. 明确步骤的人工验证。

验证应覆盖：

- 正常输入。
- 无效输入。
- 数据缺失。
- 外部依赖不可用。
- 关键输出格式。
- 失败提示和退出码。

不允许生成看似完整但不可验证的结果。

## 证据口径

小改动完成时，至少说明：

- 修改了哪些文件。
- 关键行为变化是什么。
- 运行了哪些验证。
- 哪些没有验证。
- 剩余风险是什么。
- 是否提交和推送。未经授权不得提交或推送。

接口或端到端验证应尽量提供入口、关键字段、下游位置、只读回查结果和日志证据。前端验证应尽量提供页面入口、操作路径、截图或可复现步骤、console error 和 Network 状态。

## 错误恢复流程

发生命令失败、测试失败、误操作或修改破坏现状时：

1. 停止扩大操作范围。
2. 记录当前状态：分支、HEAD、工作区状态、已执行命令、已修改文件、是否写入数据。
3. 分析失败原因，区分事实、推断和未验证风险。
4. 给出可选恢复方案和代价，由用户决定涉及回滚、删除或覆盖的操作。

未经明确授权，不得使用：

- `git reset`
- `git reset --hard`
- `git checkout <commit> -- <path>`
- `git restore`
- `git revert`
- `git clean -f`
- `git branch -D`

不得删除“看起来错误”的文件，不得回退用户已有改动。

## Git 纪律

- 修改前查看工作区状态。
- 不回退用户已有改动。
- 不把无关文件混入提交。
- 未经授权不暂存、不提交、不推送。
- 如用户要求提交，提交前说明将包含哪些文件。
- 提交信息应描述用户可理解的行为变化，而不是只写“update”。

## 密钥与敏感信息

- 不读取或输出 `.env`、`.env.*`、`credentials.json`、`*.pem`、`*.key`、token、连接串、密码的真实值。
- 必须读取配置时，只引用键名或脱敏值。
- 不得把密钥写入对话、日志、测试数据、文档示例或临时文件。
- 不得硬编码凭证。
- 发现疑似密钥已暂存或将被提交时，必须停止并报告。

## 临时产物

- 一次性排查脚本、临时 SQL、临时测试记录默认不提交。
- 可复用验证脚本可以放入 `scripts/` 或 `tmp/`，但必须有明确用途。
- 复杂 AI 协作过程材料默认不进入仓库。
- 需要沉淀的结论应整理进项目文档，而不是保存原始对话。
- 清理临时产物前必须确认不会删除用户文件。

## 字符集与平台

- 中文 Markdown 按 UTF-8 读写。
- Windows PowerShell 读取中文文件时优先使用 `Get-Content -Encoding UTF8`。
- 命令示例优先给出 Windows PowerShell 版本；如项目跨平台，再补充 bash 版本。
- 不硬编码本机绝对路径、端口、IP 或用户目录，除非文档明确说明只是本地示例。
